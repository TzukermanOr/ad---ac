מבוא ל-Domain Controller ו-Active Directory

השעה הראשונה - יסודות. נבנה את התשתית התיאורטית שעליה הכל יושב.

1.1 - מה זה דומיין (Domain) בכלל?

לפני שנדבר על Domain Controller, חייבים להבין מה זה Domain (דומיין).

דומיין הוא קבוצה של מחשבים ומשתמשים שמנוהלים יחד תחת אותם כללי אבטחה ומדיניות. תחשבו על זה כמו על "ארגון אחד גדול" - בית ספר, חברה, בית חולים - שבו יש מנהל מערכת אחד שקובע מי יכול להיכנס, מאיפה, ולאן.

💡
דוגמה מהחיים

בחברה עם 500 עובדים, אם אין דומיין - תצטרך לפתוח חשבון משתמש בכל מחשב בנפרד (500 × מספר המחשבים). עם דומיין - יוצרים את המשתמש פעם אחת ב-DC, והוא יכול להתחבר מכל מחשב בארגון.

1.2 - מה זה Domain Controller (DC)?

Domain Controller (בקיצור DC) הוא ה"מוח" של הדומיין. זה שרת ווינדוס מיוחד שמותקן עליו תפקיד (Role) בשם Active Directory Domain Services (AD DS).

🔐

אימות (Authentication)

בודק את שם המשתמש והסיסמה של כל מי שמנסה להיכנס לדומיין.

🎫

הרשאות (Authorization)

קובע למה כל משתמש מותר לגשת ולמה לא.

📋

מדיניות (Policies)

מחיל GPO - מדיניות קבוצתית על המשתמשים והמחשבים.

🗄

מאגר מידע

שומר את כל המידע על משתמשים, קבוצות, מחשבים ו-OUs במאגר הנקרא NTDS.DIT.

⚠
חשוב לדעת

DC הוא לא סתם "שרת". זה תפקיד (Role) שמותקן על שרת ווינדוס. כל מחשב Windows Server יכול להפוך ל-DC, אבל זה תהליך שצריך לבצע במפורש (Promote).

1.3 - מה זה Active Directory (AD)?

Active Directory זה השם של מערכת הקטלוג של מיקרוסופט - המקום שבו שמורים כל ה"אובייקטים" (objects) של הדומיין: משתמשים, מחשבים, קבוצות, מדפסות, וכו'.

תחשבו על AD בתור "ספריית טלפונים חכמה" של הארגון, רק שבמקום מספרי טלפון יש בה:

  • 👤 רשימת כל המשתמשים והפרטים שלהם (שם, מחלקה, אימייל, סיסמה מוצפנת)
  • 💻 רשימת כל המחשבים בארגון
  • 👥 קבוצות (לדוגמה: "אנשי IT", "מחלקת שיווק")
  • 🖨 מדפסות, שרתים, ושירותים אחרים
  • 📁 OUs (יחידות ארגוניות - נדבר עליהן בהמשך)
🎓
הקשר בין DC ל-AD

AD DS = השירות.
DC = השרת שמריץ את השירות.
AD = הקטלוג עצמו (המידע ש-DC שומר ומנהל).

1.4 - הכלים של AD DS - סקירה ראשונית

ברגע שמתקינים את התפקיד AD DS על שרת והופכים אותו ל-DC, מותקנים אוטומטית מספר כלי ניהול. אלה הכלים שתעבוד איתם 95% מהזמן:

הכלי השם המלא בשביל מה משתמשים
ADUC Active Directory Users and Computers הכלי הקלאסי. ניהול משתמשים, קבוצות, מחשבים ו-OUs.
ADAC Active Directory Administrative Center גרסה מתקדמת של ADUC עם תמיכה ב-PowerShell.
AD Domains and Trusts Active Directory Domains and Trusts ניהול אמון (Trust) בין דומיינים ו-Forests שונים.
AD Sites and Services Active Directory Sites and Services ניהול אתרים פיזיים וקישוריות בין מיקומים.
AD Module for PowerShell Active Directory Module for Windows PowerShell הפעלת פקודות AD מ-PowerShell - אוטומציה.

1.6 - היררכיה: Forest, Tree, Domain

לפני שנמשיך, חייבים להבין את ההיררכיה של AD. יש שלושה מושגים שמבלבלים את כולם בהתחלה:

היררכיה של AD: Forest מכיל Trees, Trees מכילים Domains, Domains מכילים OUs.

  • Domain (דומיין) - היחידה הבסיסית. לדוגמה: hackeru.com
  • Tree (עץ) - אוסף של דומיינים שחולקים אותו namespace (אותו שם בסיס). לדוגמה: hackeru.com, il.hackeru.com, us.hackeru.com כולם באותו Tree.
  • Forest (יער) - היחידה הגדולה ביותר. אוסף של עצים שיכולים להיות עם namespaces שונים (לדוגמה hackeru.com + cyberu.com) אבל חולקים schema אחד וקטלוג גלובלי אחד.

ארכיטקטורה ו-Entities של Active Directory

השעה השנייה - איך AD בנוי מבפנים: NTDS.DIT, Global Catalog, ואת כל סוגי האובייקטים.

2.1 - NTDS.DIT - מאגר הנתונים של AD

כל המידע של Active Directory - כל המשתמשים, הסיסמאות, הקבוצות, הכל - שמור בקובץ אחד בשם NTDS.DIT. זה למעשה בסיס הנתונים של AD.

הקובץ נמצא במיקום הזה (קבוע ב-Windows Server):

File Location 
C:\Windows\NTDS\ntds.dit

# או באמצעות משתנה סביבה:
%systemroot%\ntds\ntds.dit

בתיקייה הזו יש כמה קבצים, וכל אחד מהם משחק תפקיד:

הקובץ התפקיד
ntds.dit הקובץ הראשי - מכיל את כל האובייקטים והנתונים
edb.log קובץ ה-Transaction Log - שינויים שעוד לא נכתבו לקובץ הראשי
edb.chk Checkpoint - מסמן עד איפה כתבנו כבר ל-DIT
temp.edb קובץ זמני המשמש במהלך transactions
🚨
קריטי מבחינת אבטחה!

NTDS.DIT מכיל hashes של כל הסיסמאות בדומיין. תוקפים שמצליחים להגיע לקובץ הזה (לדוגמה דרך VSS shadow copy או DCSync) - יכולים לחלץ את ה-hashes ולעשות Pass-the-Hash על כל חשבון בדומיין, כולל Domain Admin. זה בדיוק מה שעושים כלים כמו secretsdump.py מה-Impacket.

איך לראות את הקובץ?

פתח את File Explorer

קיצור: Win + E

נווט ל-C:\Windows\NTDS

הקלד את הנתיב בשורת הכתובת, או נווט ידנית.

הפעל הצגת קבצים מוסתרים

בלשונית View, סמן Hidden items.

תראה את כל הקבצים

אבל לא תוכל לקרוא או להעתיק את ntds.dit כי זה נעול ע"י השירות.

2.2 - Global Catalog (GC) - הקטלוג הגלובלי

בארגון עם דומיין יחיד זה לא קריטי. אבל כשיש כמה דומיינים ב-Forest (לדוגמה il.hackeru.com ו-us.hackeru.com), איך מחפשים משתמש בלי לדעת באיזה דומיין הוא יושב?

התשובה: Global Catalog. זה אינדקס מצומצם של כל האובייקטים בכל הדומיינים ב-Forest, ששמור על DCs מסוימים שנקראים Global Catalog Servers.

📚
חשוב להבין

ה-GC לא מכיל את כל התכונות של כל אובייקט - רק את הפופולריות (שם, אימייל, חברות בקבוצות, וכו'). זה "partial representation" - ייצוג חלקי. למידע מלא צריך לפנות לדומיין של האובייקט.

למה משתמשים ב-GC?

  • 🔍 חיפוש מהיר של אובייקטים בכל ה-Forest
  • 🔐 אימות login בדומיינים אחרים
  • 📧 פתיחת Outlook ועדכון Global Address List
  • 👥 בדיקת חברות ב-Universal Groups

איך בודקים אם DC הוא Global Catalog?

PowerShell 
# הצג את כל ה-GCs בפורסט
Get-ADForest | Select-Object -ExpandProperty GlobalCatalogs

# בדוק DC ספציפי
Get-ADDomainController -Identity "DC01" | Select-Object Name, IsGlobalCatalog

2.3 - הארכיטקטורה הלוגית של AD

"ארכיטקטורה לוגית" זה איך AD מאורגן מבחינה רעיונית - לא איפה השרתים נמצאים פיזית. הארכיטקטורה הזו גמישה ויש להחליט עליה לפני שמקימים את הדומיין.

דוגמה לארכיטקטורה לוגית - דומיין אחד מחולק ל-OUs לפי מחלקות, כל OU עם GPO משלו.

היופי במבנה הזה הוא שכל OU יכול לקבל מדיניות (GPO) משלו. למשל, אנשי ה-IT מקבלים זכויות נרחבות, בעוד שאנשי המכירות מקבלים גישה ל-CRM אבל ה-USB שלהם חסום למניעת דליפת מידע.

2.4 - ה-Entities של AD - 4 סוגי האובייקטים העיקריים

ב-AD יש 4 סוגי אובייקטים עיקריים. כל מה שאתה רואה ב-ADUC הוא אחד מהם (או וריאציה שלהם):

👤

Users (משתמשים)

חשבונות של אנשים שיכולים להתחבר לדומיין. לכל משתמש יש שם, סיסמה, ופרטים נוספים.

מזהה ייחודי: SID, UPN, SamAccountName

💻

Computers (מחשבים)

כל מחשב שמחובר לדומיין מקבל "חשבון מחשב" משלו. גם הוא מתאמת מול ה-DC.

מזהה ייחודי: שם מחשב מסתיים ב-$ (לדוגמה PC1$)

👥

Groups (קבוצות)

אוסף של משתמשים ו/או מחשבים. נותנים הרשאות לקבוצה במקום למשתמש בודד.

2 סוגים: Security Group, Distribution Group

📁

OUs (Organizational Units)

"תיקיות" שמקבצות אובייקטים יחד. משמשות גם להחלת GPOs ולהפרדת מחלקות.

חשוב: אובייקט יכול להיות רק ב-OU אחד בכל פעם.

⚠
הבדל קריטי: OU vs. Container

OU זה לא אותו דבר כמו Container! ב-ADUC ייראו דומה, אבל: על OU אפשר להחיל GPO, על Container - לא. ה-Containers הם המובנים מראש כמו Users, Computers, Builtin.

2.5 - Containers ו-OUs המובנים

ברגע שמקימים דומיין חדש, AD יוצר אוטומטית כמה Containers ו-OUs ברירת מחדל. חשוב להכיר כל אחד:

השם סוג תפקיד
Builtin Container קבוצות מובנות עם הרשאות מנהליות (Administrators, Backup Operators, וכו')
Computers Container מיקום ברירת מחדל למחשבים שמצטרפים לדומיין
Domain Controllers OU OU מיוחד עבור כל ה-DCs. ה-OU הראשון שנוצר בדומיין!
ForeignSecurityPrincipals Container מאחסן הפניות לאובייקטים מ-Forests אחרים שמורשים
Managed Service Accounts Container חשבונות שירות מנוהלים (gMSA, MSA) - חשבונות עם סיסמה שמתחלפת אוטומטית
Users Container מיקום ברירת מחדל למשתמשים וקבוצות חדשים
🎯
Best Practice

אל תשאיר משתמשים ב-Container "Users"! בנה מבנה OUs מסודר (למשל לפי מחלקות) והעבר את המשתמשים אליו. ככה תוכל להחיל GPOs בקלות. ה-Container "Users" לא תומך ב-GPOs.

עריכת מאפייני המשתמש (Properties)

אחרי שיצרת את המשתמש, אפשר לערוך את כל המאפיינים שלו. קליק ימני על המשתמש ← Properties. תופיע חלונית עם המון לשוניות. הכי חשובות:

הלשונית למה היא משמשת
General שם פרטי, שם משפחה, תיאור, אימייל, טלפון
Account שם המשתמש, UPN, שעות התחברות מותרות, ממחשבים אילו, תוקף החשבון
Profile נתיב פרופיל, drive ממופה, סקריפט login
Member Of רשימת הקבוצות שהמשתמש שייך אליהן + הוספה/הסרה
Organization תפקיד, מחלקה, מנהל ישיר
Dial-in הגדרות חיוג מרחוק (VPN, RAS)

לשונית מוסתרת ששווה לדעת: Attribute Editor

כברירת מחדל, הלשונית Attribute Editor חבויה. כדי לראות אותה: ב-ADUC, לחץ View → Advanced Features. עכשיו תיפתח לשונית שמראה את כל המאפיינים של האובייקט - כולל SID, GUID, ועוד מאות שדות. שימושי במיוחד לפן-טסטים.

3.4 - יצירת משתמש - PowerShell

אותה פעולה, עם פקודה אחת:

PowerShell - New-ADUser 
New-ADUser `
    -Name "John Doe" `
    -GivenName "John" `
    -Surname "Doe" `
    -SamAccountName "j.doe" `
    -UserPrincipalName "j.doe@hackeru.local" `
    -Path "OU=IT-Department,DC=hackeru,DC=local" `
    -AccountPassword (Read-Host "Enter password" -AsSecureString) `
    -Enabled $true `
    -ChangePasswordAtLogon $true

פירוק הפקודה - שדה אחר שדה

הפרמטרהסבר
-Nameהשם המוצג ב-AD. חייב להיות ייחודי בתוך ה-OU.
-GivenNameשם פרטי
-Surnameשם משפחה
-SamAccountNameשם הכניסה למערכת (pre-Windows 2000). מוגבל ל-20 תווים.
-UserPrincipalNameשם הכניסה המודרני בפורמט email. לדוגמה user@domain
-Pathאיפה ה-OU/Container ב-AD. בפורמט DN - Distinguished Name.
-AccountPasswordהסיסמה - חייבת להיות מסוג SecureString.
-Enabled$true = החשבון פעיל, $false = מנוטרל.
📌
מה זה DN (Distinguished Name)?

DN הוא הכתובת המלאה של אובייקט ב-AD. הוא נכתב מימין לשמאל בהיררכיה:
CN=John Doe,OU=IT,DC=hackeru,DC=local
כלומר: יוזר John Doe → בתוך OU בשם IT → בתוך הדומיין hackeru.local.
CN = Common Name, OU = Organizational Unit, DC = Domain Component.

פקודות נוספות לניהול משתמשים

PowerShell - User Management 
# שליפת משתמש לפי שם
Get-ADUser -Identity "j.doe"

# שליפה עם כל המאפיינים
Get-ADUser -Identity "j.doe" -Properties *

# שליפת כל המשתמשים שמשהו בשם שלהם מכיל "doe"
Get-ADUser -Filter "Name -like '*doe*'"

# איפוס סיסמה
Set-ADAccountPassword -Identity "j.doe" -Reset `
    -NewPassword (ConvertTo-SecureString "NewP@ss123!" -AsPlainText -Force)

# נטרול חשבון
Disable-ADAccount -Identity "j.doe"

# הפעלת חשבון
Enable-ADAccount -Identity "j.doe"

# מחיקת משתמש
Remove-ADUser -Identity "j.doe" -Confirm:$false

# העברה ל-OU אחר
Move-ADObject -Identity "CN=John Doe,OU=IT,DC=hackeru,DC=local" `
    -TargetPath "OU=Sales,DC=hackeru,DC=local"

3.6 - קבוצות הרשאות מובנות (Built-in Privilege Groups)

AD יוצר אוטומטית כמה קבוצות עם הרשאות מנהליות מובנות. חיוני להכיר אותן - הן מהיעדים העיקריים של תוקפים.

הקבוצה היכן נמצאת מה ההרשאות סיכון
Enterprise Admins Users שליטה מלאה בכל ה-Forest, על כל הדומיינים 🔴 קריטי
Domain Admins Users שליטה מלאה בדומיין הספציפי 🔴 קריטי
Schema Admins Users שינוי ה-Schema (מבנה הנתונים) של AD 🔴 קריטי
Administrators Builtin הרשאות מנהליות ב-DC 🟠 גבוה
Server Operators Builtin תחזוקה של DCs (לא יצירת אובייקטים) 🟠 גבוה
Backup Operators Builtin גיבוי ושחזור של DCs (יכול לקרוא כל קובץ!) 🟠 גבוה
Account Operators Builtin ניהול משתמשים וקבוצות (לא privileged) 🟡 בינוני
Domain Users Users כל המשתמשים בדומיין נכנסים אליה אוטומטית 🟢 נמוך
Everyone (Implicit) כל המשתמשים המאומתים + Guests 🟢 נמוך
🎯
פרספקטיבת פן-טסטר

ב-Pentest, המטרה הסופית של רוב המתקפות על דומיין היא להגיע לחבר Domain Admins או Enterprise Admins. עם הרשאות כאלה, אתה למעשה "Game Over" - יש לך שליטה מלאה. כלים כמו BloodHound בודקים בדיוק את הנתיבים האפשריים להגיע לקבוצות הללו.

5.1 - מה קורה כשמחשב מתחבר לדומיין?

כשלוחצים Ctrl + Alt + Delete ומקלידים סיסמה, מאחורי הקלעים קורים הרבה מאוד שלבים תוך שניות. בואו נבין אותם.

5.2 - הסבר מפורט של כל שלב

שלב 1: המחשב נדלק - DNS Query

ברגע שהמחשב מקבל IP, הוא שולח שאילתת DNS מסוג SRV record כדי למצוא DC.
הוא מחפש: _ldap._tcp.dc._msdcs.hackeru.local

📌 לכן חיוני שהמחשב יקבל את ה-DC כשרת DNS שלו. אם DNS שגוי - הלקוח לעולם לא ימצא את הדומיין.

שלב 2: ה-DC מחזיר את כתובתו

שרת ה-DNS (שעל גבי ה-DC עצמו!) מחזיר את כתובת ה-IP של DC זמין. אם יש כמה DCs - הלקוח יבחר את הקרוב ביותר על פי Sites and Services.

שלב 3: אימות המחשב (Computer Authentication)

זכרו - גם למחשב יש "חשבון" ב-AD! המחשב מאמת את עצמו מול ה-DC באמצעות הסיסמה המוצפנת שלו (שמתחלפת אוטומטית כל 30 יום).
זה קורה לפני שהמשתמש מקליד שם משתמש!

שלב 4: החלת Computer GPOs

ה-DC שולח למחשב את ה-GPOs הרלוונטיים לרמת המחשב. דברים כמו: הגדרות חומת אש, רשתות אסורות, הגדרות update, וכו'. החלק הזה נטען לפני מסך ההתחברות.

שלב 5: המשתמש מקליד שם וסיסמה

המשתמש לוחץ Ctrl + Alt + Del, מקליד שם משתמש וסיסמה.
המחשב שולח ל-DC בקשת אימות Kerberos AS-REQ (Authentication Service Request).

🔐 הסיסמה לא נשלחת כפי שהיא! היא מוצפנת באמצעות hash. ה-DC כבר יודע את ה-hash הצפוי, ומשווה.

שלב 6: ה-DC מחזיר TGT

אם הסיסמה נכונה, ה-DC (שמשמש גם כ-KDC - Key Distribution Center) מחזיר TGT - Ticket Granting Ticket. זה כרטיס מוצפן שאומר "המשתמש הזה הזדהה בהצלחה".
ה-TGT תקף כברירת מחדל ל-10 שעות.

שלב 7: החלת User GPOs

עכשיו ה-DC שולח את ה-GPOs של המשתמש. דברים כמו: הגדרות שולחן עבודה, drives ממופים, סקריפטי login, מדפסות זמינות. החלק הזה נטען אחרי ההתחברות, לפני שמופיע השולחן עבודה.

שלב 8-9: גישה לשירותים (Kerberos TGS)

כשהמשתמש מנסה לגשת לשירות (לדוגמה תיקייה משותפת על שרת קבצים):
• המחשב שולח את ה-TGT ל-DC ומבקש TGS - Ticket Granting Service לאותו שירות.
• ה-DC מחזיר TGS מוצפן בסיסמה של חשבון השרת.
• המחשב מציג את ה-TGS לשרת. השרת מאמת ונותן גישה.

🎯
קשור לפן-טסטים: התקפות על Kerberos

רוב המתקפות המודרניות על AD מנצלות חולשות בתהליך הזה:
• Kerberoasting - בקשת TGS לחשבון שירות וניסיון לפצח את ההצפנה offline.
• AS-REP Roasting - חשבונות בלי "Pre-authentication" מאפשרים לקבל AS-REP ולפצח אותו.
• Pass-the-Ticket - גניבת TGT ושימוש בו ממחשב אחר.
• Golden Ticket - יצירת TGT מזויף עם hash של חשבון krbtgt - שליטה מלאה בדומיין.

5.4 - תרגיל מעשי סופי: בניית סביבה מאפס

התרגיל הסופי הוא תרחיש מקצה לקצה. נדמיין שאנחנו אדמינים במשרד עורכי דין קטן שצריך לבנות סביבה. נבנה את הכל ב-PowerShell.

Complete Setup Script - LawFirm.ps1 
# ============================================
# הקמת AD למשרד עורכי דין - תרחיש מעשי
# דרישות: דומיין lawfirm.local, 3 מחלקות,
# 9 משתמשים, קבוצות הרשאה לכל מחלקה
# ============================================

Import-Module ActiveDirectory

# שלב 1: יצירת מבנה OUs
New-ADOrganizationalUnit -Name "Departments" -Path "DC=lawfirm,DC=local"

$departments = "Litigation", "Corporate", "IT"
foreach ($dept in $departments) {
    New-ADOrganizationalUnit `
        -Name $dept `
        -Path "OU=Departments,DC=lawfirm,DC=local" `
        -ProtectedFromAccidentalDeletion $true
}

# שלב 2: יצירת קבוצות לכל מחלקה
foreach ($dept in $departments) {
    New-ADGroup `
        -Name "$dept-Users" `
        -GroupCategory Security `
        -GroupScope Global `
        -Path "OU=$dept,OU=Departments,DC=lawfirm,DC=local" `
        -Description "All users in $dept department"
}

# שלב 3: יצירת 3 משתמשים בכל מחלקה
$password = ConvertTo-SecureString "TempP@ss2025!" -AsPlainText -Force

$users = @(
    @{First="Alice"; Last="Cohen"; Dept="Litigation"},
    @{First="Bob"; Last="Levy"; Dept="Litigation"},
    @{First="Carol"; Last="Avraham"; Dept="Litigation"},
    @{First="David"; Last="Mizrahi"; Dept="Corporate"},
    @{First="Eva"; Last="Peretz"; Dept="Corporate"},
    @{First="Frank"; Last="Yosef"; Dept="Corporate"},
    @{First="Gila"; Last="Bar"; Dept="IT"},
    @{First="Hagai"; Last="Shalom"; Dept="IT"},
    @{First="Iris"; Last="Ben-David"; Dept="IT"}
)

foreach ($u in $users) {
    $sam = "$($u.First.Substring(0,1).ToLower())$($u.Last.ToLower())"

    New-ADUser `
        -Name "$($u.First) $($u.Last)" `
        -GivenName $u.First `
        -Surname $u.Last `
        -SamAccountName $sam `
        -UserPrincipalName "$sam@lawfirm.local" `
        -Path "OU=$($u.Dept),OU=Departments,DC=lawfirm,DC=local" `
        -AccountPassword $password `
        -Enabled $true `
        -ChangePasswordAtLogon $true `
        -Department $u.Dept

    # הוסף לקבוצה של המחלקה
    Add-ADGroupMember -Identity "$($u.Dept)-Users" -Members $sam
}

# שלב 4: הענקת הרשאות מיוחדות ל-IT
Add-ADGroupMember -Identity "Domain Admins" -Members "gbar"

Write-Host "✓ Setup complete!" -ForegroundColor Green
Get-ADUser -Filter * -SearchBase "OU=Departments,DC=lawfirm,DC=local" |
    Format-Table Name, SamAccountName, Department
💡
למה זה עצום?

הסקריפט הזה יוצר 9 משתמשים, 3 OUs ו-3 קבוצות + הקצאה לקבוצות, ב-30 שניות.
ב-GUI זה היה לוקח 20-30 דקות של עבודה חוזרת ומשעממת. זה הכוח של PowerShell.

✓ סיכום השעה החמישית

  1. הרץ klist במחשב שלך - תראה את ה-tickets הנוכחיים.
  2. הרץ gpresult /r - תראה אילו GPOs הוחלו עליך.
  3. נסה לזהות בכל שלב של הסכמה - איזה פרוטוקול משחק בו (DNS/LDAP/Kerberos/SMB).
  4. הרץ את הסקריפט שלמעלה במעבדה. ודא שהכל עובד.
